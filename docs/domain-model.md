# Life OS — Domain Model v0.1

## Статус документа

Документ разделяет подтверждённый минимальный доменный срез и понятия, которые пока остаются гипотезами. Только раздел **Confirmed Domain Slice: Daily Planning** разрешено использовать как основание для первой реализации.

## Confirmed Domain Slice: Daily Planning

### Day

`Day` представляет календарную дату пользователя.

Минимальные данные:

- `id` — UUID;
- `date` — календарная дата без времени и timezone.

Правила:

- на одну календарную дату существует не более одного `Day`;
- дата определяется клиентом в локальном часовом поясе пользователя и передаётся backend явно;
- backend не определяет текущую дату на основании часового пояса сервера.

В границах первого vertical slice `Day` рассматривается как агрегат, контролирующий добавление действий. Это локальное решение первого среза, а не окончательная агрегатная граница всей системы Life OS.

### Action

`Action` — конкретное действие, запланированное пользователем на определённый `Day`.

Минимальные данные:

- `id` — UUID;
- `day_id` — идентификатор существующего `Day`;
- `title` — название действия;
- `created_at` — момент создания в UTC.

Правила:

- в первом срезе `Action` принадлежит ровно одному `Day`;
- `Action` не существует без `Day`;
- создание `Action` означает добавление нового действия непосредственно в выбранный день;
- `Action` не является `Event`, `Habit`, `Goal` или `Project`;
- `title` после нормализации пробелов не может быть пустым;
- действия отображаются в порядке `created_at`, затем `id`.

## First Vertical Slice

Первый пользовательский сценарий:

1. Пользователь открывает выбранный календарный день.
2. Система получает существующий `Day` или создаёт новый.
3. Пользователь добавляет `Action`.
4. После повторного чтения `Action` отображается в том же `Day`.

В первый срез не входят:

- completion status;
- editing;
- deletion;
- moving actions between days;
- custom ordering;
- priority;
- description;
- tags;
- deadlines;
- duration;
- `Event`;
- `Habit`;
- `Goal`;
- `Project`;
- `Review`;
- AI;
- authentication;
- integrations.

## Hypotheses and Deferred Concepts

### Основная цепочка-гипотеза

Identity → Domain → Vision → Goal → Project → Outcome → Action → Day

Смысл связей в этой цепочке, кардинальности, правила владения и жизненные циклы пока не подтверждены. Цепочка не должна использоваться как готовая схема данных.

### Поддерживающие понятия-гипотезы

- Event
- Habit
- Metric
- Review
- Decision
- Resource

Эти понятия сохраняются как направления дальнейшего исследования. Они не являются подтверждёнными сущностями и не должны порождать таблицы, API или модули до появления конкретного пользовательского сценария и согласованных инвариантов.

## Отложенные решения

До следующих срезов не фиксируются:

- полная долгосрочная иерархия;
- универсальные связи между сущностями;
- окончательная роль `Day` во всей модели;
- статусы и жизненные циклы действий;
- теги, классификации и произвольные связи;
- границы агрегатов за пределами первого vertical slice.
