# Life OS — Architecture v0.1

## 1. Предлагаемая архитектура

На первом этапе Life OS предлагается строить как локальное веб-приложение в формате модульного монолита:

- frontend отвечает за пользовательский интерфейс и сценарии работы;
- backend предоставляет API, выполняет прикладные операции и обеспечивает целостность данных;
- реляционная база данных является единым источником истины;
- доменные модули разделены логически, но разворачиваются как одно приложение.

Внутри backend применяется слоистая архитектура с направлением зависимостей от внешних слоев к внутренним:

1. API — прием запросов и формирование ответов.
2. Application — пользовательские сценарии и координация операций.
3. Domain — сущности, правила и инварианты.
4. Infrastructure — база данных, файловая система и другие технические адаптеры.

Архитектура не закрепляет конкретные фреймворки. Их следует выбрать отдельно до создания технического каркаса.

## 2. Почему архитектура подходит для Life OS

Life OS — персональное локальное приложение с единой связанной предметной областью. На начальном этапе ему не нужны независимые сервисы и распределенная инфраструктура. Модульный монолит:

- сохраняет простоту запуска, разработки и резервного копирования;
- позволяет поддерживать согласованность связанных данных;
- соответствует принципу одного источника истины;
- дает возможность развивать отдельные части системы без преждевременного разделения на сервисы;
- учитывает важность ежедневного пользовательского сценария, не предопределяя его техническую реализацию;
- оставляет границы модулей достаточно четкими для последующего выделения частей системы, если это станет оправданно.

## 4. Backend-архитектура

Backend организуется как набор логически изолированных компонентов внутри одного процесса. Архитектурный документ не определяет состав доменных сущностей и не фиксирует границы доменных модулей. Актуальная доменная модель и состав сущностей определяются только документом `docs/domain-model.md` и будут уточняться отдельными решениями до реализации бизнес-функций.

Основные правила backend:

- API не содержит бизнес-правил;
- пользовательский сценарий оформляется как отдельная прикладная операция;
- доменный слой не зависит от HTTP, базы данных и frontend;
- работа с хранилищем скрыта за интерфейсами репозиториев;
- транзакции задаются на уровне завершенного пользовательского сценария;
- проверка входного формата отделяется от проверки доменных инвариантов;
- ошибки имеют стабильные машинные коды и понятные сообщения.

Ежедневный пользовательский сценарий является важной частью продуктовой концепции. Его техническая реализация, правила владения данными и агрегатные границы пока не утверждены и должны быть определены отдельным решением до реализации.

## 5. Frontend-архитектура

Frontend предлагается организовать по пользовательским сценариям и предметным сущностям:

- `app/` — запуск приложения, маршрутизация, глобальная конфигурация и композиция зависимостей;
- `pages/` — страницы верхнего уровня;
- `features/` — законченные действия пользователя;
- `entities/` — представление доменных сущностей и связанная с ними клиентская логика;
- `shared/` — переиспользуемые элементы интерфейса, утилиты и API-клиент без доменной специализации.

Интерфейс должен поддерживать важный для продукта ежедневный сценарий, не закрепляя на уровне архитектуры конкретную структуру экрана или доменные границы. Долгосрочный контекст отображается через связанные сценарии, а не копируется в отдельное клиентское хранилище как второй источник истины.

Frontend хранит только состояние интерфейса и кэш серверных данных. Долговременное состояние сохраняется через backend. Формы должны явно отображать состояния загрузки, отсутствия данных, ошибок и успешного сохранения.

## 6. Взаимодействие frontend и backend

Frontend взаимодействует с backend через локальный HTTP API с JSON-представлением данных. API версионируется с первой публичной версии, например префиксом `/api/v1`.

Для контракта взаимодействия предлагается:

- описывать запросы, ответы и ошибки в машиночитаемой спецификации API;
- использовать стабильные идентификаторы сущностей;
- передавать даты и время в однозначном стандартном формате;
- не раскрывать структуру таблиц базы данных через API;
- проверять совместимость контракта автоматическими тестами;
- выполнять изменения данных явными командами, а чтение — отдельными запросами.

На первом этапе достаточно синхронного request-response взаимодействия. Событийная шина, WebSocket и фоновые очереди добавляются только при появлении подтвержденного сценария.

## 7. Хранение данных

Для первой локальной версии предлагается реляционная база SQLite:

- она не требует отдельного сервера;
- база размещается в одном локальном файле;
- транзакции помогают сохранять согласованность связанной модели;
- данные легко переносить и резервировать;
- схему можно развивать управляемыми миграциями.

Принципы хранения:

- база данных является единственным источником долговременного состояния;
- изменение схемы выполняется только версионируемыми миграциями;
- доменные сущности не зависят от структуры ORM или таблиц;
- внешние ключи и ограничения базы дополняют доменные проверки;
- время хранится в едином формате, а правила часовых поясов согласуются отдельно;
- пользовательская база, резервные копии и временные файлы не попадают в Git;
- формат экспорта и стратегия резервного копирования определяются до работы с реальными данными.

Если в будущем появятся подтвержденные ограничения SQLite, инфраструктурный слой должен позволить перейти на другую реляционную СУБД без изменения доменной логики и API-сценариев.

## 8. Расширяемость системы

Расширяемость обеспечивается не количеством абстракций, а устойчивыми границами:

- каждый новый сценарий добавляется в соответствующий модуль;
- доменные правила остаются независимыми от интерфейса и хранилища;
- API меняется обратно совместимо либо получает новую версию;
- доступ к техническим ресурсам осуществляется через заменяемые адаптеры;
- межмодульные зависимости направлены явно и не образуют циклов;
- общие компоненты выделяются только после появления реального повторного использования;
- миграции сохраняют историю развития данных;
- модульные, интеграционные и контрактные тесты защищают границы системы.

Новые возможности не должны концентрировать всю логику приложения в одном универсальном компоненте. Ответственность распределяется согласно доменной модели и отдельно согласованным границам.

## 9. Архитектурные принципы

1. **Local First.** Основные сценарии и пользовательские данные должны быть доступны локально без обязательной внешней инфраструктуры.
2. **Documentation First.** Существенные решения сначала фиксируются в соответствующей документации в рамках отдельной явно поставленной задачи, а затем реализуются.
3. **Domain First.** Технические решения следуют согласованной доменной модели, а не формируют ее неявно через код или схему базы данных.
4. **API First.** Взаимодействие frontend и backend начинается с явного, версионируемого и проверяемого контракта.
5. **Single Source of Truth.** Для каждого вида данных определяется один авторитетный источник, а производные представления не становятся независимыми копиями.
6. **Small Iterations.** Система развивается небольшими завершенными изменениями с понятным результатом и проверкой.
7. **Simplicity.** Выбирается самое простое решение, которое удовлетворяет текущим подтвержденным требованиям и сохраняет ясные границы.
8. **Testability.** Компоненты и зависимости проектируются так, чтобы правила, сценарии и интеграции можно было проверять изолированно.

## 10. Принципы разработки

1. Разрабатывать небольшими проверяемыми итерациями.
2. Не добавлять функциональность до появления согласованного сценария.
3. Не изменять доменную модель без явного решения и обновления документации.
4. Сохранять один источник истины для каждой сущности.
5. Сначала фиксировать контракт и правило, затем выбирать техническую реализацию.
6. Разделять доменную логику, прикладные сценарии и инфраструктурный код.
7. Предпочитать простые решения и избегать преждевременной универсальности.
8. Проверять доменные правила модульными тестами, работу с базой — интеграционными, API — контрактными.
9. Применять автоматическое форматирование, статический анализ и единые команды проверки.
10. Не хранить секреты, пользовательские данные и локальные артефакты в репозитории.
11. Обновлять защищенную проектную документацию только в рамках отдельной явно сформулированной задачи.
12. Поддерживать возможность локального запуска и восстановления данных без внешней инфраструктуры.

## Решения перед созданием технического каркаса

До начала реализации необходимо согласовать:

- языки и фреймворки frontend и backend;
- способ запуска одного или двух процессов в локальной среде;
- точные границы backend-модулей;
- правила идентификаторов, дат, времени и часовых поясов;
- формат спецификации API и генерацию типов клиента;
- инструмент миграций и доступа к базе;
- минимальный набор проверок в первой итерации;
- правила размещения, резервного копирования и экспорта локальной базы.
